# Agentic Runtime Core v02

## Objective

Define the operational core for a governed multi-agent system that manages a website's daily work without sacrificing identity or educational quality.

## Core model

The runtime is composed of five layers:

1. Constitutional layer
2. Harness / supervisor layer
3. Specialist agents
4. Critic and evaluation layer
5. Memory and learning layer

## Why this model

Generic agent systems optimize for throughput. This framework instead optimizes for:
- trust
- distinctiveness
- teacher usefulness where education is in scope
- pedagogical integrity
- operational reliability
- controlled growth
- verifiability

## Runtime role map

### Supervisor
- Harness

### Planner
- Intake & Triage
- Content Strategy for multi-step content work

### Operators / doers
- Technical Health
- SEO & Crawl
- Content Production
- Monetization & Merchandising

### Guardians
- Brand Steward
- Pedagogical Steward
- Governance, Security & Approval

### Critics
- Brand Steward
- Pedagogical Steward
- Analytics & Evaluation

### Presenter
- the final typed output, report artifact, or publish recommendation

## Core runtime claims

- The harness is the only orchestration authority.
- Agents are narrow and replaceable.
- Agents communicate through typed artifacts and queues.
- Guardian and critic loops are mandatory where scope requires them.
- Evaluation determines learning promotion.
- Verification matters as much as generation quality.

## Core control loop

Signal → classify → plan → route → execute → validate → critique → decide → writeback → evaluate → promote or regress

## Runtime constraints

- No content publish path can bypass brand and pedagogy gates.
- No risky write can bypass governance.
- No agent can change its own charter.
- Market intelligence cannot be used to rewrite site voice.
- Self-healing must preserve auditability.
- Core architecture changes require stronger review than leaf work.
