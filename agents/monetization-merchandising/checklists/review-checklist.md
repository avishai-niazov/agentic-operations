# Review Checklist — Monetization & Merchandising

Use this checklist before recommending any monetization change. Every item must be answered explicitly. If any item fails, the recommendation is held, not sent for approval.

## 1. Surface and scope

- [ ] The affected surface is specific (single page, template class, hub, sitewide).
- [ ] The scope is the smallest that can test the hypothesis.

## 2. User-risk tradeoff

- [ ] The tradeoff between revenue and user experience is stated in plain terms.
- [ ] Teacher usefulness is explicitly considered where the surface is used by teachers.
- [ ] Trust impact is addressed (no disguised ads, no misleading affiliate framing).

## 3. Speed and performance

- [ ] The change's effect on page speed is estimated or measured.
- [ ] Speed-critical surfaces (hubs, top landing pages) are flagged.
- [ ] If speed risk is non-trivial, Technical Health has been consulted or will be.

## 4. Brand and pedagogy alignment

- [ ] The change respects `knowledge/content/non_negotiables_v01.md` and brand voice.
- [ ] The change does not turn editorial into sales.
- [ ] The change does not inject competitor-style promotional patterns.

## 5. Rollback definition

- [ ] The rollback trigger is named — the observable signal that requires reversal.
- [ ] The rollback path is defined and reversible within a named time window.
- [ ] "Leave it and watch" is not an acceptable rollback plan for sitewide changes.

## 6. Experiment design (if experiment)

- [ ] Hypothesis is stated.
- [ ] Success metric is named (RPM, CTR, revenue, or combined) alongside a guardrail metric (speed, engagement, trust).
- [ ] Duration and sample are defined before starting.
- [ ] Segmentation avoids unintentional harm to specific audiences (e.g., mobile, teacher).

## 7. Governance readiness

- [ ] The recommendation is sized correctly for the approval matrix.
- [ ] Sitewide or template-level changes are routed to Governance & Security.
- [ ] Disclosure implications (affiliate, sponsored, commerce) are named.

## 8. Prohibited-action check

- [ ] No aggressive-ad push.
- [ ] No bypass of governance approval.
- [ ] No UX-damaging change without a rollback path.
